# LLM-guided research path

By default Agnitra runs the **specialist optimization path** — a hard-coded sequence of proven passes (TF32, SDPA, static cache, `torch.compile`, optional INT8 weight-only quantization). The specialist path makes no network calls and is the right choice for production workloads.

This page documents the **research path**: an optional LLM-guided pipeline (`enable_rl=True`, `use_specialist=False`) that asks an LLM to suggest kernel-tuning parameters. The research path exists so you can explore optimization patterns the specialists don't yet cover. **Do not enable it for production traffic** — outputs are non-deterministic and add latency for every optimization decision.

## When to use the research path

- Investigating whether a fused RoPE+RMSNorm kernel is worth implementing for an architecture
- Exploring quantization regimes the standard specialists don't yet support
- Benchmarking against academic papers that describe novel optimizations

## How to invoke

```python
import agnitra

result = agnitra.optimize(
    model,
    input_shape=(1, 512),
    enable_rl=True,
    use_specialist=False,    # required: routes off the specialist path
)
```

Both flags are required. `use_specialist=True` (the default) bypasses the research path even if `enable_rl=True`.

## LLM backends

Agnitra picks a backend based on which API keys / processes are available, in this order:

1. **OpenAI** if `OPENAI_API_KEY` is set and the `openai` extra is installed (`pip install "agnitra[openai]"`).
2. **Codex CLI** if it is on the user's `PATH`.
3. **Ollama** if a local Ollama server is running on `AGNITRA_OLLAMA_URL` (default `http://localhost:11434`).
4. **No-op** — the optimizer returns the unchanged baseline if no backend is reachable.

Override the auto-selection by setting `LLMOptimizerConfig(backend="openai" | "ollama" | "codex" | "none")`.

## What the LLM sees

The optimizer sends a JSON payload describing:

- Architecture fingerprint (`model_type`, key dims)
- Per-layer profiler telemetry (CUDA times, tensor shapes, memory)
- Hardware target (e.g. `H100`)

It does NOT send model weights, input data, or generated outputs.

## What the LLM returns

A structured tuning preset — TF32 toggle, attention implementation choice, compile mode, etc. The optimizer applies the preset, validates the output against the baseline (same drift safety net as the specialist path), and reverts if the suggestion regresses.

## Caveats

- **Non-deterministic.** The same model can receive different optimizations across runs.
- **Adds latency.** A network round-trip per optimization decision. Cache hits via the architecture fingerprint mitigate this for repeat workloads.
- **Costs money.** OpenAI calls bill against your API account. Ollama is free but slower.

For production, use `use_specialist=True` (the default).
